Materialized View 매번 주문 테이블을 JOIN·SUM하지 않고 일별 GMV를 저장Daily Report
CREATE MATERIALIZED VIEW IF NOT EXISTS ecom.mv_daily_gmv AS
SELECT DATE_TRUNC('day', o.order_ts) AS day,
       SUM(oi.line_total) AS gmv
FROM ecom.orders o
JOIN ecom.order_items oi ON oi.order_id = o.order_id
WHERE o.order_status IN ('paid','shipped','delivered')
GROUP BY DATE_TRUNC('day', o.order_ts);

CREATE UNIQUE INDEX ux_mv_daily_gmv_day
ON ecom.mv_daily_gmv(day);

REFRESH MATERIALIZED VIEW ecom.mv_daily_gmv;
갱신 전략 — 데이터 변경 주기에 맞춰 매일 오후 3시 일반 REFRESH를 실행한다. 운영 중 조회 차단을 피해야 하면 고유 인덱스 생성 후 CONCURRENTLY 갱신을 적용한다.

Nested Loop

소수 고객을 기본 키로 조회하는 Q10의 최종 JOIN처럼 외부 결과가 작고 내부 인덱스가 있을 때 적합하다.

Hash Join

Q1~Q5, Q8~Q10처럼 주문상품 다수를 동등 조건으로 결합할 때 반복 탐색을 피하고 대량 처리에 유리하다.

Merge Join

양쪽 입력이 조인 키로 정렬됐거나 정렬 비용을 이후 연산에서도 활용할 수 있을 때 경쟁력이 생긴다.

JOIN 선택의 핵심

특정 JOIN 방식을 무조건 강제하는 것이 아니라 통계, 실제 행 수, 인덱스, 메모리와 정렬 상태를 개선해 옵티마이저가 적합한 계획을 고르도록 해야 한다.
